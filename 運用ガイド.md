# 運用ガイド

セットアップ完了後、日常的に行う運用作業の手順書です。
詳細な解説は `GitHub_Desktop_マニュアル.md` を参照。

---

## ケース別 早見表

### 🟢 ケース1：問題を追加する

最も頻度の高い作業です。

1. テキストエディタで `data/questions.json` を開く
2. `questions` 配列の末尾、`]` の直前に問題オブジェクトを追加（カンマ忘れ注意）
3. `questionDataVersion` を今日の日付に更新（例：`"2026.6.1"`）
4. `totalCount` の数値も問題数に合わせて更新
5. GitHub Desktop → Commit → Push

**問題オブジェクトのテンプレート：**
```json
{
  "id": "amashi_35_1",
  "examType": "amashi",
  "year": 35,
  "questionNo": 1,
  "subject": "医療概論",
  "section": "AM",
  "questionText": "問題文をここに記載",
  "choices": ["選択肢1", "選択肢2", "選択肢3", "選択肢4"],
  "correctAnswers": [2],
  "explanation": "解説をここに記載",
  "sourceUrl": "",
  "sourcePdf": ""
}
```

**フィールドの意味：**

| フィールド | 内容 | 例 |
|---|---|---|
| `id` | 問題の一意なID。`<試験種別>_<年>_<問題番号>` 形式推奨 | `"amashi_35_1"` |
| `examType` | 試験種別。`amashi`（あマ指）/ `hari_kyu`（はり師きゅう師） | `"amashi"` |
| `year` | 試験回。数字または文字列 | `35` または `"あマ指　第35回予想"` |
| `questionNo` | 問題番号 | `1` |
| `subject` | 科目名 | `"医療概論"` |
| `section` | 時間帯。`AM` / `PM` | `"AM"` |
| `questionText` | 問題文 | |
| `choices` | 選択肢配列（通常4つ） | `["...", "...", "...", "..."]` |
| `correctAnswers` | 正解の選択肢番号配列（1始まり） | `[2]`（2番が正解の場合）|
| `explanation` | 解説 | |
| `sourceUrl` | 出典URL（任意） | `""` |
| `sourcePdf` | 出典PDF（任意） | `""` |

---

### 🟡 ケース2：問題を修正する

1. `data/questions.json` を開く
2. 該当する問題オブジェクトを編集
3. `questionDataVersion` を更新
4. Commit → Push

---

### 🟡 ケース3：問題を削除する

1. `data/questions.json` を開く
2. 該当する問題オブジェクト全体（`{ ... },` の塊）を削除
   - 削除した結果、配列末尾の問題のカンマが余らないよう注意
3. `questionDataVersion` を更新
4. `totalCount` を減らした数に更新
5. Commit → Push

**注意：** 削除した問題のIDで学習履歴を持っていた利用者の進捗は、その問題分は残ったまま参照できなくなります。

---

### 🟠 ケース4：アプリ本体を修正する

1. `index.html` を編集
2. ファイル先頭付近の `const APP_VERSION = '2026.5.18';` を新しい日付に
3. Commit → Push

---

### 🔴 ケース5：間違ったコミットを取り消したい

1. GitHub Desktopの「History」タブを開く
2. 取り消したいコミットを右クリック
3. 「Revert this commit」を選択
4. 自動で取り消しコミットが作られる
5. Push

## 月次・定期で気にしておくとよいこと

| 頻度 | 作業 |
|---|---|
| 都度 | 問題追加・修正後、必ず `questionDataVersion` を更新 |
| 月1回 | GitHub の Settings → Pages で公開状態を確認 |
| 月1回 | 友人に「バックアップ取った？」と一声かける |
| 大幅変更時 | 友人に事前連絡（「明日問題追加します」など） |

## 友人側で困ったときの対処（管理者用メモ）

| 友人からの相談 | 確認・回答 |
|---|---|
| 「問題が更新されない」 | ブラウザのスーパーリロード（Ctrl + Shift + R / iPhone Safariは設定→Safari→履歴とWebサイトデータを消去） |
| 「進捗が消えた」 | バックアップJSONがあれば「📤 取込」で復元可能。なければ復元不可 |
| 「機種変更したい」 | 旧端末で「📥 出力」→ JSONをメール等で新端末に送る → 新端末で「📤 取込」 |
| 「7日触らないと消える?」 | iOSのみ。週1起動 or 定期バックアップで回避 |
| 「URLが開けない」 | GitHub Pagesは稀にメンテで一時停止。数時間後に再アクセス |

## バージョン番号の付け方ルール（推奨）

| バージョン | 形式 | 更新タイミング |
|---|---|---|
| `APP_VERSION` | `YYYY.M.D` | index.html を変更した日 |
| `questionDataVersion` | `YYYY.M.D` | questions.json を変更した日 |

両方とも同じ日付形式ですが、それぞれ別の場所にあり、別々に更新します。

**例：**
- 2026年6月1日に問題を3問追加 → `questionDataVersion` のみ `"2026.6.1"` に
- 2026年6月5日にアプリのUIを修正 → `APP_VERSION` のみ `'2026.6.5'` に
- 両方同時に変更したら、両方とも同じ日付に

## チェックリスト：プッシュ前の最終確認

- [ ] `questionDataVersion` を更新した（問題データを変えたとき）
- [ ] `APP_VERSION` を更新した（index.htmlを変えたとき）
- [ ] JSONの構文エラーがないか（カンマ・括弧）
- [ ] `totalCount` の数値が `questions` 配列の要素数と一致している
- [ ] Commit Summaryに分かりやすい説明を書いた

すべてOKなら Push。
