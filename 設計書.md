# 設計書：アプリと問題データの分離運用

## 1. 改修の目的

国試対策アプリの問題データを追加・更新するたびにHTML全体を差し替えるのは手間が大きい。
GitHubでアプリ本体と問題データを分け、問題データだけ更新できるようにする。

加えて、学習履歴のバックアップを促す機能を追加する。

## 2. 改修前の構造

```
study_app.html  ← 1ファイルにHTML/CSS/JS/問題データすべて
  └── BUNDLED_QUESTIONS = [...]  ← JS内に直書き
```

問題1問追加するだけでも HTMLファイル全体（約130KB）を再配布する必要があった。

## 3. 改修後の構造

```
リポジトリ
├── index.html              ← アプリ本体（外部fetch化）
├── data/
│   └── questions.json      ← 問題データ（独立ファイル）
└── .nojekyll               ← GitHub Pages用
```

GitHub Pagesで公開し、利用者はブラウザでアクセスするだけ。
問題データを更新したいときは `data/questions.json` だけ編集すればよい。

## 4. 改修内容（4点）

### 4-1. 問題データの外部化

| 項目 | 改修前 | 改修後 |
|---|---|---|
| 配置場所 | `index.html` 内のJS定数 `BUNDLED_QUESTIONS` | `data/questions.json` |
| 読み込みタイミング | スクリプト評価時に即時利用可 | `init()` 内で `fetch()` |
| 失敗時の挙動 | （該当なし） | 既存IndexedDBデータで継続。完全に空ならエラー画面 |

問題データJSONの構造：

```json
{
  "schemaVersion": "1.0",
  "questionDataVersion": "2026.5.18",
  "lastUpdated": "2026-05-18",
  "description": "あん摩マッサージ指圧師・はり師きゅう師国家試験 対策問題集",
  "totalCount": 10,
  "questions": [ { ...問題オブジェクト... }, ... ]
}
```

問題オブジェクトの構造は改修前と完全に同じ（`id, examType, year, questionNo, subject, section, questionText, choices, correctAnswers, explanation, sourceUrl, sourcePdf`）。

### 4-2. バージョン管理の二重化

| バージョン | 管理対象 | 比較先 |
|---|---|---|
| `APP_VERSION` | アプリ本体（index.html）の更新日 | IndexedDB `settings.appVersion` |
| `questionDataVersion` | 問題データJSONの更新日 | IndexedDB `settings.questionDataVersion` |

どちらか一方でも新しければ、`BUNDLED_QUESTIONS` をIndexedDBに再投入する。

**運用ルール：**
- アプリ本体に変更を加えたら `index.html` の `APP_VERSION` を更新（例：`2026.5.18` → `2026.6.1`）
- 問題データに変更を加えたら `data/questions.json` の `questionDataVersion` を更新

### 4-3. 5日経過バックアップ通知

最終バックアップから5日以上経過していたら、起動時に上部バナーで通知する。

| 状態 | 動作 |
|---|---|
| 初回起動 | `firstSeenAt` を記録。即時通知はしない |
| 初回起動から5日以上経過・バックアップ未実施 | 「📥 学習履歴のバックアップをおすすめします」バナー表示 |
| バックアップ実施済み・5日以上経過 | 「📥 バックアップから○日経過」バナー表示 |
| バックアップ実施直後 | バナー非表示 |

バナーには「今すぐバックアップ」「あとで」ボタン。「今すぐ」を押すと既存の `exportData()` を呼び、`lastBackupAt` を更新。

### 4-4. インポート・エクスポート機能

**変更なし。既存実装をそのまま利用。**
- エクスポート：`exportData()` → `study_backup_YYYYMMDD.json` を端末にダウンロード
- インポート：`importData()` → 学習履歴・メモ・設定を上書き復元

ただし `exportData()` の末尾で `lastBackupAt` を記録する1行を追加（5日通知のため）。

## 6. データ保存場所

| データ | 保存先 | 共有範囲 |
|---|---|---|
| アプリ本体（index.html） | GitHubリポジトリ | 公開（誰でも閲覧可） |
| 問題データ（questions.json） | GitHubリポジトリ | 公開（誰でも閲覧可） |
| 学習履歴・進捗 | 各端末のIndexedDB | 端末ローカルのみ |
| メモ | 各端末のIndexedDB | 端末ローカルのみ |
| 設定 | 各端末のIndexedDB | 端末ローカルのみ |
| バックアップJSON | 端末のダウンロードフォルダ | 利用者が管理 |

## 7. 既知の制約

| 制約 | 影響 | 対策 |
|---|---|---|
| Public リポジトリ | 問題データが第三者から閲覧可能 | 友人内利用のため許容 |
| iOS Safari 7日ルール | 7日以上開かないとIndexedDBが削除される可能性 | 5日バックアップ通知で対策 |
| デバイス間で進捗共有不可 | 機種変更で進捗消失 | エクスポート/インポートで対応 |
| ネットワーク必須 | オフラインでは初回起動不可 | 一度起動すればキャッシュで継続可能（ブラウザ依存） |

## 8. 動作確認手順

1. ローカルで `python3 -m http.server` 等のWebサーバを起動
2. ブラウザで `http://localhost:8000/` を開く
3. 「問題データを更新しました」のトーストが表示される（初回）
4. 各メニューが動作することを確認
5. 設定 → 「📥 出力」でバックアップJSONがダウンロードされる
6. ブラウザの IndexedDB を一度クリア → 「📤 取込」でJSONを選択 → 復元されることを確認

※ `file://` プロトコルでは `fetch()` がブロックされるため、必ずHTTP経由で開くこと。
